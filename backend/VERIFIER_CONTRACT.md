# Python Verifier Output Contract (Version 1)

This document defines the output interface/contract of the Python verifier (`verify_image.py`) consumed by the Node.js backend (`ComplaintDecisionService`).

---

## Contract Version Info
- **Current Version:** `1`
- **Field name:** `contract_version`
- **Value type:** `integer`
- **Significance:** Version `1` represents a clean separation where the Python verifier outputs raw classification and detection signals, and business policies (warning status, blocks, retries) are computed exclusively by the Node.js backend.

---

## Output Schema JSON Structure

A canonical version 1 payload looks like this:

```json
{
  "decision": "FOOD_MATCH_PLAUSIBLE",
  "reason": "Verification passed: Plausible food match detected (burger) and insect-like object detected.",
  "confidence": 0.95,
  "detected_objects": ["insect/bug"],
  "issue_support_level": "issue_supported_strong",
  "food_match_result": "food_match_plausible",
  "face_detected": false,
  "predicted_label": "burger",
  "prediction_confidence": 0.95,
  "checks": {
    "files_valid": true,
    "quality_ok": true,
    "screen_recap_suspected": false,
    "metadata_ok": true
  },
  "ordered_item": "burger",
  "complaint_type": "FOREIGN_OBJECT",
  "reasoning": "Evidence strongly supports the complaint. Detected insect/bug.",
  "contract_version": 1
}
```

## Field Specifications

### Required Fields (Must be present in version 1 output)

1. **`decision`** (`string`)
   - **Allowed values:** `REJECT`, `REJECT_FACE_PRESENT`, `REJECT_NON_FOOD`, `FOOD_MISMATCH`, `FOOD_MATCH_PLAUSIBLE`, `UNCERTAIN`
   - **Description:** The primary verifier classification outcome.

2. **`reason`** or **`reasoning`** (`string`)
   - **Description:** Explanatory text generated by the verifier models or sanity checks describing the output.

3. **`food_match_result`** (`string`)
   - **Allowed values:** `food_match_plausible`, `food_mismatch`, `uncertain`
   - **Required status:** Required for non-hard-rejections (i.e. when `decision` is not `REJECT`, `REJECT_FACE_PRESENT`, or `REJECT_NON_FOOD`).
   - **Description:** Outcome of expected food category matching logic.

### Optional / Signal Fields

1. **`issue_support_level`** (`string`)
   - **Allowed values:** `issue_supported_strong`, `issue_supported_weak`, `issue_not_visible`, `issue_uncertain`, `fraud_suspected`
   - **Default fallback:** `issue_uncertain` (or `fraud_suspected` for hard rejects).
   - **Description:** CLIP anomaly check support classification level indicating how strongly the evidence visualizes the reported issue.

2. **`confidence`** (`number` between `0.0` and `1.0`)
   - **Description:** Overall confidence score of the classification.

3. **`detected_objects`** (`array` of `string`)
   - **Description:** Raw labels of anomaly objects detected (e.g. `['insect/bug']`, `['hair']`). Defaults to `[]` if not present.

4. **`face_detected`** (`boolean`)
   - **Description:** True if a human face was detected in any of the uploaded images.

5. **`predicted_label`** (`string`)
   - **Description:** The detected food category name (e.g., `burger`, `pizza`, etc.).

6. **`prediction_confidence`** (`number`)
   - **Description:** Confidence score associated with `predicted_label`.

7. **`checks`** (`object`)
   - **Structure:**
     - `files_valid` (`boolean`)
     - `quality_ok` (`boolean`)
     - `screen_recap_suspected` (`boolean`)
     - `metadata_ok` (`boolean`)
   - **Description:** Individual integrity check statuses.

8. **`ordered_item`** (`string`)
   - **Description:** The expected food item label passed to the verifier.

9. **`complaint_type`** (`string`)
   - **Description:** The complaint reason category.

## Deprecation & Compatibility Notes

### Deprecated Fields (Do NOT produce in Python)
The following fields were previously included in the verifier output but are now **deprecated** and have been removed from version 1 outputs:
- **`recommended_status`**
- **`recommended_warning_state`**

### Node.js Legacy Payload Tolerance
- To prevent deployment mismatches, the Node.js backend continues to accept older payloads containing the deprecated fields.
- If these fields are encountered, the backend ignores their values for policy decisions and prints a deprecation warning (`console.warn`) urging a python-layer update.
